# Premium Web Design Skill

Use this skill for any homepage, landing page, SaaS UI, portfolio page, pricing section, testimonial section, hero section, dashboard mockup, or screenshot-to-code task.

## Core Rule

Do not produce generic AI-looking web design.

The goal is premium, conversion-focused, production-quality web design that looks intentionally designed, not assembled from default cards and gradients.

When a screenshot or mockup is provided, treat it as the visual source of truth. Match layout, spacing, hierarchy, proportions, mood, and visual density before making creative changes.

## Design Standard

The final result should feel closer to:

- Linear
- Vercel
- Framer
- Stripe
- Mercury
- Ramp
- Superhuman
- Webflow premium templates
- high-end SaaS product studios

It should not feel like:

- a generic AI agency template
- default Tailwind cards
- random gradients
- oversized empty sections
- disconnected pill buttons
- fake startup clutter
- inconsistent icon/card systems

## Visual Priorities

Prioritize in this order:

1. Layout fidelity
2. Spacing and rhythm
3. Typography hierarchy
4. Alignment
5. Visual balance
6. Conversion clarity
7. Responsive behavior
8. Motion/decoration only after the above are correct

Never add decorative elements that make the layout less clear.

## Screenshot-to-Code Rules

When implementing from a screenshot:

1. First describe the screenshot structure:
   - page width
   - background
   - navbar
   - grid layout
   - major sections
   - visual hierarchy
   - spacing
   - card sizes
   - button placement
   - image/mockup placement
2. Then implement the layout.
3. Then run the app locally.
4. Then take a screenshot at:
   - desktop: 1440x1000
   - laptop: 1280x900
   - mobile: 390x844
5. Compare the rendered screenshot to the reference.
6. Fix obvious mismatches before reporting completion.

Do not report done after only writing code.

## 1stStep.ai Visual System

Use this design direction unless the user explicitly provides a different reference:

- Subtle grid texture
- Blue, cyan, and purple gradient accents
- Premium glass cards
- Thin glowing borders
- Soft radial background glow
- Large confident typography
- Clean CTA hierarchy
- Dashboard/product mockups that look functional
- Rounded corners: 20px to 32px
- High contrast text
- Minimal but sharp icons
- No muddy low-contrast text
- No washed-out dashboard previews
- No overly tall dead-space sections

## Typography Rules

Hero headlines:

- Desktop: large, bold, tight line-height
- Target size: 64px to 84px depending on layout
- Line-height: 0.95 to 1.05
- Do not stack one or two words per line unless the reference does that intentionally
- Highlight only the most important phrase with gradient text

Section headlines:

- 40px to 56px desktop
- Strong contrast
- Clear visual rhythm

Body text:

- 17px to 20px desktop
- Comfortable line height
- Muted but readable

Small labels:

- Uppercase tracking is okay
- Must remain readable
- Do not overuse labels

## Layout Rules

Use max-width containers:

- 1280px to 1440px for major sections
- Centered on page
- Consistent horizontal padding

Desktop hero:

- Two-column layout when the reference has one
- Left column: headline, copy, CTAs, trust points
- Right column: product/dashboard visual
- Avoid huge empty gaps
- Above-the-fold should feel full but not cramped

Cards:

- Equal heights when grouped
- Consistent padding
- Consistent border radius
- Clear internal hierarchy
- No random card widths
- No cramped text
- No button overlapping text

Buttons:

- Primary CTA should be visually dominant
- Secondary CTA should be quieter but clear
- Use consistent height, padding, and border radius
- Never let button text overlap other content

## Dashboard Mockup Rules

If building a product mockup in HTML/CSS:

It should look like a real product, not a placeholder.

Include:

- sidebar
- top label
- metrics
- chart/progress area
- next steps/checklist
- realistic labels
- glowing border
- subtle depth
- strong contrast

For 1stStep.ai hero dashboard, use:

- MVP Score: 87
- Recommended Path
- Estimated Timeline: 2-4 weeks
- Next Steps:
  - Define MVP Scope
  - Design Workflow
  - Build & Integrate
  - Test & Refine
  - Launch & Improve

Do not make dashboard text too tiny or blurry.

## Pricing Card Rules

Pricing cards should answer:

- Who is this for?
- What does it solve?
- What do they get?
- What is the starting price?
- What should they click next?

Use stronger card labels:

- Website + Lead System
- Internal Tool or Dashboard
- MVP App Build

Pricing cards should include:

- icon
- category badge
- title
- starting price
- short outcome-driven description
- best-for list
- CTA

Do not use generic descriptions like "most flexible" unless it helps the buyer decide.

## Testimonial / Google Review Rules

For Google review sections:

- Use the Google "G" logo where appropriate.
- Put the Google logo above or beside "Google Reviews."
- Use real person profile photos if actual reviewers have public profile photos available.
- If real profile photos are not available, use polished initials or neutral avatars. Do not invent fake real people images unless the page is clearly a mockup.
- The "Read or leave a review" button must not overlap checklist text.
- Testimonial cards need clean spacing, readable text, and star ratings.
- Use "View on Google" with Google G icon inside the link row.
- Do not fake review count or average rating unless provided.

## Before / After Section Rules

Before/after sections should sell transformation.

Before:

- unclear first version
- manual work spread across tools
- no conversion path
- scope keeps growing

After:

- focused first version
- connected workflow
- clear path to action
- launch-ready roadmap

Do not make this section look like a spreadsheet.

Use strong icons, contrast, and short human-readable language.

## Copywriting Rules

Write like a premium operator, not a hype marketer.

Use:

- practical
- focused
- launch
- workflow
- first version
- build path
- scope
- dashboard
- automation
- conversion path
- lead system
- internal tool

Avoid:

- revolutionary
- unleash
- unlock your potential
- supercharge everything
- AI-powered magic
- guaranteed results
- vague buzzwords

Keep copy simple and direct.

## Responsiveness

Every design must be checked at:

- 1440px desktop
- 1280px laptop
- 768px tablet
- 390px mobile

Mobile:

- stack columns
- reduce headline size
- preserve CTA visibility
- avoid horizontal overflow
- hide or simplify decorative floating cards if needed
- keep text readable

## Accessibility

Maintain:

- high contrast
- semantic headings
- meaningful button/link text
- alt text for images
- visible focus states
- no text embedded in images if it needs to be indexed/read
- no tiny unreadable UI text

## Technical Rules

Do not break:

- routes
- API routes
- forms
- booking links
- App Idea Checker links
- pricing anchors
- portfolio links
- env vars
- analytics
- lead capture behavior

Do not add:

- paid API calls
- new external dependencies unless necessary
- large animation libraries unless already installed
- fake customer metrics
- fake guaranteed results

Prefer:

- existing components
- Tailwind utility classes
- reusable local components
- HTML/CSS mockups over flat mockup images when possible

## Done Means

A task is not done until:

1. Code is implemented.
2. Desktop screenshot is taken.
3. Mobile screenshot is taken.
4. Visual mismatch is reviewed.
5. Obvious spacing/layout/design issues are fixed.
6. Lint passes, or the absence of a lint script is reported.
7. Build passes.

Report back with:

- files changed
- visual changes made
- links preserved
- lint result
- build result
- screenshot comparison notes
